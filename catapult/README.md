# Катапульта на ESP32

Прикладний проєкт: керована катапульта. Механіку (плече, рама, спуск) друкуємо на 3D-принтері,
**електроніку й прошивку пишемо самі**. Це не «одна кнопка → кидок», а невелика мехатронна
система з повним циклом і станами.

> Статус: 🟢 конструкцію обрано — плата **ESP32-C6-Zero**, механізм **мотор-натяг + сервозасувка**.
> Це **навчальний проєкт: прошивку пишеш ти сам.** Цей README — спільний довідник
> (конструкція, піни, застороги), а **не готовий код**.
> Тулчейн готовий: **ESP-IDF v5.5.4 + Qt Creator**. Проєкт прошивки — `firmware/`
> (порожній каркас від `idf.py create-project`; **логіку пишеш ти**). Як збирати/заливати — §7.

---

## 1. Катапульта як мехатронна система

Будь-яка катапульта (хоч римський онагр, хоч наша друкована) працює за одним циклом —
**накопич енергію → утримай → націль → відпусти → перезаряди**. У прошивці це лягає на
**скінченний автомат** (state machine):

```
        ┌─────────┐  кнопка «звести»   ┌──────────┐  досягнуто межі  ┌────────┐
        │  IDLE   │ ─────────────────▶ │ COCKING  │ ───────────────▶ │ ARMED  │
        │ (спокій)│                    │ (натяг)  │                  │(заряд.)│
        └─────────┘                    └──────────┘                  └────────┘
             ▲                                                            │
             │                                                            │ кнопка «вогонь»
             │ перезаряд завершено                                        ▼
        ┌──────────┐         автоповернення             ┌──────────────────────┐
        │ RESETTING│ ◀───────────────────────────────── │       FIRING          │
        │(повернути)│                                    │ (спуск засувки, кидок)│
        └──────────┘                                    └──────────────────────┘
```

Чому саме автомат, а не «if-и»:
- кожна фаза має **свої дозволені дії** (у `ARMED` не можна знову тягнути мотор);
- переходи прив'язані до **подій** (кнопка, кінцевий вимикач, таймер), а не до `delay()`;
- легко додати **безпеку**: блокування подвійного пострілу, тайм-аут натягу, аварійний стоп.

## 2. Інженерні підзадачі (що саме доведеться запрограмувати)

| Підзадача | Суть | Розділи курсу |
|---|---|---|
| Привід натягу | крутити мотор/сервопривід, що зводить плече | PWM/ЦАП (25), MOSFET/H-міст (12) |
| Відчуття «зведено» | кінцевий вимикач / Hall / лік кроків | GPIO (22), переривання (23) |
| Спуск (засувка) | сервопривід відмикає клямку точно в мить пострілу | таймери (24) |
| Кнопки керування | «звести» / «вогонь» з придушенням дребезгу | GPIO (22) |
| Скінченний автомат | стани + переходи без блокування | модель виконання (27) |
| Індикація | LED/звук/дисплей: який зараз стан | — |
| (далі) Наведення | сервопривід повороту рами | сервопривід, ПІД (34) |
| (далі) Дистанційний пуск | кнопка → Wi-Fi/BLE з телефона | Wi-Fi/BLE (38) |

## 3. Що в нас є під це (з інвентаря)

**Плати ESP32:** ESP32-S3 SuperMini (USB-C), ESP32-S3-Pico, ESP32-C6-Zero, ESP32-CAM.
**Приводи:** N20 мотор-редуктор ×2, сервопривід SG90 (9 г), сильніший сервопривід HD-1370A,
кроковий 28BYJ-48 + ULN2003, BLDC.
**Драйвери:** MC33886 (H-міст 5 А), реле 5 В ×2.
**Датчики межі/удару:** мікро-кінцевики (тактові кнопки), Hall KY-003, геркон KY-021,
оптопереривач KY-010, віброудар KY-031.
**Індикація:** LED, пасивний/активний баззер, LCD1602 (I2C), 7-сегмент.
**Живлення:** LiPo 3S (XT60), програмований DC-DC buck-boost, 14500 Li-ion.

## 4. Рішення, які треба ухвалити (перш ніж писати код)

Оскільки деталі ще не надруковані, **механізм можна обрати першим, а друк підігнати під нього.**
Два рішення задають усю прошивку (пінмапу, драйвери, автомат):

1. **Механізм пострілу** — прямий кидок сервоприводом / мотор натягує + сервозасувка /
   кроковий натягує + сервозасувка. Визначає, чим керуємо й чи потрібен датчик межі.
2. **Плата ESP32** — визначає пінмапу, USB-прошивання, наявність камери.

## 4а. Обрана конструкція (ESP32-C6-Zero)

**Привід натягу:** N20 мотор-редуктор через H-міст MC33886. **Засувка:** сервопривід SG90.
**Датчик «зведено»:** кінцевий вимикач (тактова кнопка) у крайньому задньому положенні плеча.
**Керування:** дві кнопки (звести / вогонь). **Індикація:** онборд WS2812.

### Таблиця пінів

| Функція | GPIO | Режим / нотатки |
|---|---|---|
| MC33886 IN1 | IO0 | LEDC PWM 20 кГц · **+10 kΩ pulldown до GND** |
| MC33886 IN2 | IO1 | LEDC PWM 20 кГц · **+10 kΩ pulldown до GND** |
| SG90 сигнал | IO2 | LEDC 50 Гц (імпульс 1–2 мс) |
| Кінцевий вимикач | IO3 | `INPUT_PULLUP`, контакт до GND (активний LOW) |
| Кнопка «звести» | IO20 | `INPUT_PULLUP`, до GND |
| Кнопка «вогонь» | IO21 | `INPUT_PULLUP`, до GND |
| Резерв (баззер) | IO14 | вільний |
| RGB онборд | **IO8** | WS2812 через `led_strip` (RMT); на Waveshare C6-Zero дані на GPIO8 |

> Уникаємо strapping-пінів **IO8/IO9/IO15** та USB **IO12/IO13**.

### Живлення й земля

- **ESP32-C6-Zero** — від USB-C (або 5 В пад).
- **N20** — від програмованого buck-boost на номінал мотора → у `VM` MC33886; логіку MC33886 (`VCC`) на 3V3 або 5 В.
- **SG90** — від 5 В шини (пік струму ~600 мА; не з 3V3 плати).
- **Спільна земля** ESP32 ↔ MC33886 ↔ серво ↔ джерело сили — обов'язково.

### Чотири ключові застороги

1. ESP32 дає лише **сигнали 3.3 В**; сила йде з окремого джерела зі спільною землею.
2. **Стан пінів під час boot:** pulldown на IN1/IN2 тримає мотор вимкненим, доки не стартувала прошивка.
3. **Серво = 50 Гц PWM** (ширина імпульсу), мотор = 20 кГц PWM — різні таймери LEDC.
4. **Тайм-аут натягу → FAULT:** якщо плече не дійшло до кінцевика, мотор вимикається (захист від спалювання).

### Що доведеться підібрати, коли писатимеш прошивку

- два кути серво — «тримає клямку» і «спускає»;
- потужність натягу (PWM мотора) — момент проти гумки;
- тайм-аут натягу — скільки часу даємо на повний хід до кінцевика (запобіжник);
- напрямок обертання мотора — залежить від того, як підключені дроти N20.

## 6. Принципи прошивки (ESP-IDF)

- **жодних блокувань** у керуванні — час через `esp_timer_get_time()` / тики FreeRTOS, не busy-wait;
- точка входу `app_main()` (запускається як задача FreeRTOS); цикл/задачі організовуєш сам;
- діагностика через `ESP_LOGI/ESP_LOGW` (рівень лога — у `menuconfig`);
- спершу **safe state** в `app_main()` (засувка зачинена, мотор вимкнений), тоді логіка;
- периферія явними драйверами: `gpio`, `ledc` (PWM серво/мотора), за потреби `gptimer`.

## 7. Середовище розробки (ESP-IDF + Qt Creator)

**Встановлено:** ESP-IDF **v5.5.4** (`C:\Users\andri\esp\esp-idf`), тулчейн `riscv32-esp-elf` + OpenOCD (`~/.espressif`), Qt Creator **19**.

**Як це працює (перевірено — збірка з IDE проходить):** проєкт **сам піднімає середовище ESP-IDF** через [`firmware/idf_env.cmake`](firmware/idf_env.cmake), підключений у `CMakeLists.txt` **перед** `project.cmake`:

```cmake
include(${CMAKE_CURRENT_LIST_DIR}/idf_env.cmake OPTIONAL)   # IDF_PATH, PATH(+тулчейн), PYTHON, IDF_TARGET
include($ENV{IDF_PATH}/tools/cmake/project.cmake)
```

`idf_env.cmake` робить `set(ENV{...})` для `IDF_PATH`, тулчейна в `PATH`, `IDF_TARGET=esp32c6`, і **закріплює venv-Python** (`set(PYTHON … CACHE … FORCE)`) — інакше CMake бере системний Python без модулів ESP-IDF. `set(ENV{})` під час конфігурації успадковують дочірні процеси (компілятор, ninja, python).

**Kit у Qt Creator:** будь-який manual-кіт, але його **CMake generator має бути Ninja** (jom/NMake ESP-IDF не підтримує). CMakePresets і env-блок у кіті **не потрібні** — усе дає `idf_env.cmake`.

**Зібрати:** File → Open → `firmware/CMakeLists.txt` → обрати кіт → Configure → молоток (**Build**). Перед повторним конфігом після змін — **Build → Clear CMake Configuration**.

**Машинні шляхи** живуть лише в `idf_env.cmake` (одне місце; правити при оновленні ESP-IDF).

**Заливка:** `powershell -ExecutionPolicy Bypass -File catapult\tools\flash.ps1` — заливає те, що зібрав Qt Creator, і відкриває монітор (`Ctrl+]` — вийти). Аргументи: `-Port COM5`, `-Build ESP32_IDF-Release`. (Git Bash: `bash tools/flash.sh`. Або run-крок у Qt Creator.)

> Глухі кути (не повторювати): `sdktool` застарий для QtC19 (псує конфіг); env-блок у кіті крихкий (`IDF_PATH` не доходив → `ld: --major-image-version`); `CMakePresets.json` — QtC тримався старого кіта через `.user`. `tools/qtcreator-esp-idf.ps1` лишається як запасний.
